# v1.1 — Authenticated Grading Documentation Page (`/handleiding`)

**Status:** design draft. Not implemented. Ship in v1.1, before the
weight-customization UI lands. Schools need to understand the math
before they flex it.
**Author:** discussion 2026-05-01
**Owner:** TBD

## Why this exists

Today, when a teacher sees `Eindniveau: 2.3 / 4.0` on a student profile,
or a docent reads `code_kwaliteit: 64.1 (bayes), conf 0.08`, they have
no way to answer:

- **Where did 2.3 come from?** Six werkprocessen at four niveaus each
  → how does that aggregate?
- **What's a "Bayesian" score?** Why is the legacy score 80 but the
  Bayesian score 64?
- **What does confidence 0.08 mean?** Should I trust the radar?
- **Why did veiligheid drop from 73 to 60 between PR 2 and PR 3?**
- **Why is testen low when the student wrote 4 tests?**
- **Why did phpstan flag testing when our rubric weighs it 0%?**
- **What does the LLM actually see when it grades my student's PR?**

Without answers, school admins distrust the numbers, docenten can't
defend grades to students or parents, and weight customization (when
v1.1 ships it) is a blind exercise.

This page is the **shared mental model** that makes everything else
defensible. Required reading before a school admin tries to flex
their rubric weights — they need to understand what they're flexing.

## Audience

Three reader profiles:

1. **School admin / curriculum coordinator** (primary). Wants to
   understand the math well enough to defend it to the inspectie and
   sign off on per-vak weight changes. Will read carefully.
2. **Docent (teacher)**. Wants to understand why a grade is what it
   is so they can answer student/parent questions in 30 seconds.
3. **Curious student**. Wants to know "is this fair?" and "how do I
   improve faster?"

Tone: technical but humane. No marketing copy. Worked examples with
real numbers, not abstractions.

## Where it lives

- **URL:** `/handleiding` (authenticated only — students, teachers,
  admins all see it; viewers see a slimmer version)
- **Nav placement:** Header link "Handleiding" between existing items;

### 5. De wiskunde: van findings naar Eindniveau

This is the core. Step-by-step walkthrough using a real example:

> Student Vash heeft 4 PR's gehad. Hier is hoe de huidige Eindniveau
> 2.4 / 4.0 tot stand kwam.

#### 5.1 Per-criterium SkillObservations

Each PR's GradingSession has 6 rubric scores (one per werkproces).
Show the actual table:

| PR | code_ontwerp | code_kwaliteit | veiligheid | testen | verbetering | samenwerking |
|---|---|---|---|---|---|---|
| #1 | 2 | 2 | 1 | 1 | — | 2 |
| #2 | 2 | 3 | 3 | 2 | 2 | 3 |
| #3 | 3 | 3 | 3 | 3 | 3 | 3 |
| #4 | 2 | 3 | 3 | 2 | 2 | 3 |

Each (PR, criterion) pair becomes a `SkillObservation` row. The
"weighted score" stamped on each observation = niveau × 25 (so 1=25,
4=100).

#### 5.2 Bayesian convergence

`update_bayesian()` blends each new observation into the running
metric using a learning rate that decays as confidence grows:

```
learning_rate = max(0.05, 0.5 * (1 - confidence))
new_score = old_score * (1 - lr) + observation * lr
new_confidence = old_confidence + 0.05 * complexity_weight
```

Worked numerical example for `veiligheid`:

| Step | Observation | LR | Score | Confidence |
|---|---|---|---|---|
| start | — | — | 50.0 | 0.00 |
| PR #1 (niveau 1 → 25) | 25 | 0.50 | 37.5 | 0.05 |
| PR #2 (niveau 3 → 75) | 75 | 0.475 | 55.3 | 0.10 |
| PR #3 (niveau 3 → 75) | 75 | 0.45 | 64.2 | 0.15 |
| PR #4 (niveau 3 → 75) | 75 | 0.425 | 68.8 | 0.20 |

So `veiligheid` shows 68.8 with confidence 0.20 — even though the last
3 PRs were all niveau 3 (75). The score is honest about uncertainty:
the bad PR #1 still drags the score because we've only seen 4 data
points.

#### 5.3 Eindniveau aggregation

The teacher's `Eindniveau` widget is a **weighted average across the
6 werkprocessen**, mapped from 0-100 to 0-4:

```
weighted_score = sum(metric.bayesian_score * werkproces.weight)
                / sum(werkproces.weight)
eindniveau = weighted_score / 25
```

For the canonical Crebo weights (15+20+20+20+10+15 = 100):

```
weighted = (52 × 0.15) + (66 × 0.20) + (68.8 × 0.20)
         + (54 × 0.20) + (58 × 0.10) + (65 × 0.15)
         = 60.8 / 100
eindniveau = 60.8 / 25 = 2.43
```

So `Eindniveau: 2.4 / 4.0` ≈ "Op opleidingsniveau / Gedeeltelijk
beheerst" zone. The widget colors the niveau bar accordingly.

#### 5.4 Why 2.4 and not 2.0?

Walk through the rationale: the Bayesian score is a *prior + observed
evidence*, not a simple average. With more PRs, the score converges
toward the observed niveau. With fewer, it stays close to the
neutral 50 prior.

Anti-FAQ:
- **"My student got niveau 4 on every criterion in PR 3. Why is
  Eindniveau still 2.4?"** Because PR 1 and 2 still count. Bayesian
  weighting doesn't erase history; it dampens it as evidence
  accumulates. After ~10 niveau-4 PRs, `Eindniveau` will land near 4.
- **"My student is at 2.4 but the radar shows code_kwaliteit at 3.0.
  Inconsistency?"** No: the radar is per-werkproces; Eindniveau is
  the weighted aggregate. They measure different things.

### 7. Deterministic vs LLM findings

Two layers, different rules:

- **LLM (Haiku/Sonnet)** reads your rubric and grades against it.
  Affected by criterion weights.
- **Deterministic (ruff/eslint/phpcs/phpstan)** runs canonical rules
  regardless of rubric. Their findings show up in the recurring-
  patterns chart but don't directly affect Eindniveau if your rubric
  doesn't weight that werkproces.

Worked example: "Your rubric weights `testen` 0%. phpstan still flags
missing tests. Student sees the warning in their Code Review. The
Eindniveau widget ignores it for grading. The skill graph still
records it as a SkillObservation under TESTING category — useful for
the docent's 'are they making the same mistake repeatedly' check."

This is honest about the asymmetry. Schools that want phpstan to
respect their custom weights will need v2.

### 8. Confidence en betrouwbaarheid

Explain why scores have a confidence field:

- 0.00-0.15: preliminary. Radar fades. Eindniveau widget shows
  "voorlopig" badge.
- 0.15-0.40: enough data to be honest about direction.
- 0.40-1.00: trust the score; it's based on enough observations.

Connect to the practical recommendation: "Wacht op 5+ PR's voordat je
een student-gesprek baseert op het Eindniveau. Daarvoor zijn de
nummers indicaties, geen verdicten."

## Implementation notes

- **Static markdown rendered server-side**, not Vue routes. Markdown
  files in `frontend/src/content/handleiding/` rendered via `marked`
  or `unified` at build time. Simpler to update than Vue components,
  no risk of dashboard-style data-fetch failures, easier for
  non-engineers to edit.
- **Each section is its own anchor**. Deep links from widgets ("?"
  icons) jump to the relevant explainer.
- **Worked examples regenerated periodically** from a real demo
  cohort. The numbers in section 5 should track reality, not stale
  copy.
- **Authenticated only** but no role gate. Students benefit from
  reading section 5 ("how is my grade computed?") more than admins.

## Out of scope for v1.1

- Per-school customization of the doc text (orgs can't override the
  numbers in section 5)
- Automated worked-example regeneration from each org's data (cool
  but expensive; defer to v1.5)
- Embedded interactive calculator ("change the weights, see the
  Eindniveau change") — would be amazing for v2 weight customization
  onboarding but adds frontend complexity

## Decision log

- **Why authenticated, not public?** The page describes internal
  scoring math, prompt templates, and PII handling. School admins
  signing the contract see it; the public `/rubric` page (marketing)
  stays at the higher-level "what we grade" framing.
- **Why markdown, not Vue?** Lower update friction. A non-engineer
  can edit copy without redeploys.
- **Why ship before weight customization?** Without this doc,
  schools that flex weights are flying blind. Read first, then flex.
- **Why include the LLM prompt?** GDPR transparency. Docenten need
  to be able to answer "what does Anthropic see about my student?"
  on the spot.
- **Why include deterministic vs LLM section?** Today there's a real
  asymmetry — phpstan fires regardless of rubric weights. Hiding that
  would damage trust when a school admin discovers it.
