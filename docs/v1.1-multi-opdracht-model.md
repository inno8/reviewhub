# v1.1 — Multi-Opdracht Data Model

**Status:** design draft. Not implemented. Ship after May 7 pitch.
**Author:** discussion 2026-04-30
**Owner:** TBD

## Why this exists

v1 of LEERA binds one repo per student per cohort via
`CohortMembership.student_repo_url`. That worked for the dogfood phase
but it does not match how docenten actually run a vak.

A vak (Course) like Database runs for a year. During that year a
docent assigns multiple opdrachten (projects). Each opdracht has its
own brief, its own deadline, its own emphasis on the rubric, and
each student turns in a separate repo per opdracht. Over a year, one
vak with 25 students produces ~100 repos (4 opdrachten × 25 students).

v1 cannot represent this. v1.1 introduces an Assignment layer.

## Hierarchy

```
Organization (the school)
  └─ Cohort (klas — group of ~25 students for a year)
       └─ Course (vak — Database, Webdev, OOP — taught by one docent)
            └─ Assignment (opdracht — one project within the course)
                 └─ StudentAssignment (this student → this opdracht → this repo)
                      └─ Submission (one per PR against that repo)
                           └─ GradingSession (AI draft + docent review iteration)
```

Existing models stay; two new ones land between Course and Submission.

## New models

### `Assignment` (opdracht)

```python
class Assignment(models.Model):
    course = models.ForeignKey(Course, on_delete=models.CASCADE,
                               related_name='assignments')
    name = models.CharField(max_length=200)              # "Opdracht 3: Reporting Dashboard"
    slug = models.SlugField(max_length=120)              # for URL routing
    description = models.TextField(blank=True)           # the opdracht briefing — markdown

    # Sequencing
    order = models.PositiveIntegerField(default=0)       # display order within the course
    starts_at = models.DateField(null=True, blank=True)  # week the opdracht opens
    deadline = models.DateField(null=True, blank=True)   # inleverdatum

    # Rubric weighting overrides for this opdracht.
    # Each opdracht inherits the Course's Rubric but can boost or
    # mute specific SkillCategories. Stored as a JSON map
    # {category_name: weight_multiplier}, default {} = no overrides.
    # Example: {"PERFORMANCE": 1.5, "TESTING": 0.5} means this
    # opdracht weights performance 50% more and tests 50% less than
    # the course default — useful for a "tuning + indexing" opdracht
    # where test coverage isn't the focus.
    rubric_weight_overrides = models.JSONField(default=dict, blank=True)

    # State machine
    STATE_CHOICES = [
        ('draft', 'Draft'),         # docent is still writing the brief, hidden from students
        ('open', 'Open'),           # students can submit
        ('grading', 'Grading'),     # past deadline, docent reviewing
        ('closed', 'Closed'),       # final grades issued
        ('archived', 'Archived'),   # hidden from default views
    ]
    state = models.CharField(max_length=16, choices=STATE_CHOICES, default='draft')

    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['course', 'order', 'created_at']
        unique_together = [('course', 'slug')]
        indexes = [
            models.Index(fields=['course', 'state']),
        ]
```

### `StudentAssignment`

```python
class StudentAssignment(models.Model):
    """
    Links a student to an Assignment with the repo they're using
    for that opdracht. Replaces CohortMembership.student_repo_url
    as the webhook matcher's source of truth.
    """
    assignment = models.ForeignKey(Assignment, on_delete=models.CASCADE,
                                   related_name='student_assignments')
    student = models.ForeignKey(User, on_delete=models.CASCADE,
                                related_name='student_assignments')
    repo_url = models.URLField(max_length=500)           # canonical https URL
    repo_full_name = models.CharField(max_length=200)    # "owner/repo" — denormalized for fast webhook lookup

    # Optional link back to the StudentRepo (GitHub App install) that
    # provides the token to clone + post comments. Null when the repo
    # is connected via legacy PAT or when the App install hasn't
    # been linked yet.
    student_repo = models.ForeignKey(
        'users.StudentRepo', null=True, blank=True,
        on_delete=models.SET_NULL,
        related_name='student_assignments',
    )

    # Computed cache
    latest_submission = models.ForeignKey(
        'grading.Submission', null=True, blank=True,
        on_delete=models.SET_NULL,
        related_name='+',
    )
    current_grade = models.DecimalField(max_digits=4, decimal_places=2,
                                        null=True, blank=True)

    # Soft-delete so re-invite can restore
    removed_at = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        unique_together = [('assignment', 'student')]
        indexes = [
            models.Index(fields=['repo_full_name']),       # webhook hot path
            models.Index(fields=['student', 'removed_at']),
        ]
```

## Webhook matcher rewrite

Today (`grading/webhooks.py:_memberships_for_pr`):

```python
qs = CohortMembership.objects.filter(
    student_repo_url__icontains=repo_full_name,
    removed_at__isnull=True,
)
```

v1.1 becomes:

```python
qs = StudentAssignment.objects.filter(
    repo_full_name=repo_full_name,
    removed_at__isnull=True,
    assignment__state__in=['open', 'grading'],
).select_related('assignment', 'assignment__course', 'student')
```

For shared-repo group projects (multiple students on one repo), the
matcher returns all StudentAssignment rows pointing at that repo. The
existing `_pick_primary_author` logic still picks a primary based on
PR author, others go to `Submission.contributors`.

The Submission gets `assignment_id` instead of `course_id` directly:

## Rubric resolution

`GradingSession` resolves the effective rubric per submission as:

```
effective_rubric = course.rubric  # base
for category, weight in assignment.rubric_weight_overrides.items():
    effective_rubric.weights[category] *= weight
```

This is computed once at GradingSession creation; the AI engine
receives the resolved weights, not the raw overrides. Docenten see
the resolved rubric in the review UI with a "boosted" / "muted" badge
on overridden categories so they understand why the AI focused where
it did.

## Migration plan

Single migration `grading/migrations/0XXX_assignment_studentassignment.py`:

1. Create `Assignment` and `StudentAssignment` tables.
2. Add nullable `assignment_id` to `Submission`.
3. Data migration: for every existing `Cohort`:
   - For every `Course` in the cohort, create a single
     `Assignment(course=course, name="Opdracht 1", slug="opdracht-1",
     state="open", order=0)` — one default opdracht per course.
   - For every `CohortMembership` with `student_repo_url`, create
     a `StudentAssignment(assignment=<that course's default>,
     student=membership.student, repo_url=membership.student_repo_url,
     repo_full_name=parse(...))`.
   - For every existing `Submission`, set
     `submission.assignment_id = <its course's default opdracht>`.
4. Mark `CohortMembership.student_repo_url` as deprecated (keep
   the column for a release; drop in v1.2).

Rollback: drop the two new tables, null out `Submission.assignment_id`.
The deprecated `student_repo_url` is still populated, so v1 behavior
is intact.

## API changes

New endpoints under `/api/grading/`:

```
GET  /courses/<id>/assignments/                 list assignments in a course
POST /courses/<id>/assignments/                 create an assignment (docent only)
GET  /assignments/<id>/                         detail
PATCH /assignments/<id>/                        update brief / deadline / state
GET  /assignments/<id>/students/                list StudentAssignments in this opdracht
POST /assignments/<id>/students/                add student → repo link
DELETE /student-assignments/<id>/               soft-remove a student from an opdracht
GET  /students/<id>/assignments/                a student's active opdrachten across courses
```

Existing `/cohorts/<id>/recurring-errors/` and the snapshot/trajectory
endpoints stay; they just gain optional `?assignment_id=` filtering
so docenten can scope skill deltas to one opdracht.

## Teacher UI changes

**Course detail page** gains an "Opdrachten" tab:
- List of assignments with state badges (draft / open / grading / closed)
- "Nieuwe opdracht" button → modal with name, brief markdown editor,
  deadline picker, rubric override sliders (one per category, default 1.0)
- Each row links to the assignment detail with the student → repo grid

**Assignment detail page** is new:
- Header: name, deadline, state, "Edit" button
- Briefing tab (rendered markdown)
- Students tab: table of students × repo URL × current grade × last
  push timestamp × "Open submissions" link
- Skill movement chart scoped to this opdracht

**New review queue** filters by assignment instead of cohort.

## Student UI changes

**Dashboard** shows a "Mijn opdrachten" section instead of the current
"Active projects". Each card:
- Opdracht name (bold)
- Course name (smaller, above)
- Deadline (with countdown if < 7 days)
- Current grade or "Nog geen grading"
- Connected repo link (or "Verbind repo" CTA if missing)
- Recent feedback teaser

Click → opdracht detail with full briefing + submission history +
skill deltas earned on this opdracht.

**Connect-repo flow** changes: instead of one repo per cohort, the
student picks a repo per opdracht. The picker uses their connected
StudentRepo list (from the GitHub App install). New repo? They
either install the App on it or pick one of their already-connected
repos.

## Skill graph implications

Two new aggregations the snapshot endpoint can return:
- `per_assignment_deltas: { assignment_id: { category: delta } }`
- `per_course_deltas: { course_id: { category: delta } }`

These let docenten say "Yusuf gained +14 in SECURITY across Opdracht 2"
instead of just "+14 across the whole year." Pattern recurrence
analysis stays user-scoped (a recurring SQL injection mistake is a
recurring mistake regardless of which opdracht surfaced it).

## Out of scope for v1.1

- Per-opdracht peer review (students grading each other)
- Group opdrachten with role assignment (frontend lead, backend lead)
- Cross-cohort assignments (a single opdracht shared between klassen)
- Late submission penalties
- Plagiarism detection across assignments

These can come in v1.5+. Keep v1.1 focused on the data model that
unlocks "many opdrachten per course."

## Decision log

- **Why a separate Assignment model and not just a list of repos on
  Course?** A list-of-repos can't carry brief, deadline, state, or
  rubric overrides. Assignments need first-class lifecycle.
- **Why JSON for rubric_weight_overrides instead of a join table?**
  Overrides are sparse (most opdrachten override 0-2 categories) and
  read together as a unit when grading. JSON keeps the read pattern
  fast without a join.
- **Why nullable `assignment_id` on Submission?** Backward-compat for
  v1 rows. After backfill no rows are null but the schema permits
  legacy gracefully. Drop the nullability in v1.2.
- **Why keep `course` on Submission alongside `assignment`?** Existing
  queries that group by course (skill snapshots, weekly trends) still
  work without rewrite. Denormalized for query speed; populated from
  `assignment.course` on save.
