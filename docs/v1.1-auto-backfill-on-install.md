# v1.1 — Auto-Backfill on First Repo Install

**Status:** design draft. Not implemented. Ship in v1.1, depends on the
async draft kickoff infra (Celery/RQ + Redis) since backfill should
not block the install callback.
**Author:** discussion 2026-05-01
**Owner:** TBD

## The gap this closes

When a student installs the LEERA App on a repo with N existing
commits, none of those commits get analyzed. GitHub's `push` event
payload is incremental — it only contains commits new since the
previous push. Even a brand-new install produces no historical data.

Three concrete consequences in production:

1. **Recurring-pattern detection is blind for the first month.** A
   student who's been making the same SQL injection mistake since
   week 1 won't get flagged until the pattern recurs across 2-3
   post-install commits. Their "behavioral proof" track starts at
   zero observations.
2. **Teacher's "Sterke en zwakke plekken" radar shows 0 spokes** for
   new students until they push real PRs to LEERA-watching branches.
   Confusing for the docent at the booth: "why is my student blank?"
3. **End-of-jaar grade is unfair to early adopters** vs students who
   happened to install LEERA at semester start. A student joining
   mid-semester loses 6-12 weeks of skill graph history.

## What we already have

`grading/services/github_backfill.py` + `backfill_student_history`
management command — a manual, admin-only workflow that walks merged
PRs in a 90-day window and creates Submissions + DeterministicFindings
+ SkillObservations stamped with historical timestamps. **Layer 1
only — no LLM cost.** Designed for demo seeding, not student onboarding.

The pipes work. We just need to wire them to fire automatically when a
new StudentRepo gets created.

## Trigger surface

Two webhook events fire when a student installs/updates the App:

- `installation.created` — first-ever install of the App on the
  student's account (one-time per account)
- `installation_repositories.added` — student adds a new repo to an
  existing install (can fire multiple times)

We want the backfill to run on **both**. Each time a new `StudentRepo`
row is created, kick off a background task that backfills that
specific repo.

## Design

### 1. New model: `RepoBackfillJob`

```python
# users/models.py (or grading/models.py — pick one)

class RepoBackfillJob(models.Model):
    """
    Tracks a single auto-backfill of a StudentRepo's historical commits.
    One job per (StudentRepo, window) pair. Created when the install
    webhook fires; consumed by an async worker.
    """

    class Status(models.TextChoices):
        QUEUED   = 'queued',   'In de wachtrij'
        RUNNING  = 'running',  'Bezig'
        DONE     = 'done',     'Voltooid'
        FAILED   = 'failed',   'Mislukt'
        SKIPPED  = 'skipped',  'Overgeslagen'

    student_repo = models.ForeignKey(
        'users.StudentRepo',
        on_delete=models.CASCADE,
        related_name='backfill_jobs',
    )
    student = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name='backfill_jobs',
    )
    days_window = models.PositiveIntegerField(
        default=90,
        help_text='How many days of history to backfill.',
    )
    status = models.CharField(
        max_length=16, choices=Status.choices, default=Status.QUEUED,
    )

    # Progress tracking — surfaced in the grace-period banner UI.
    prs_found = models.PositiveIntegerField(default=0)
    prs_processed = models.PositiveIntegerField(default=0)
    prs_skipped = models.PositiveIntegerField(default=0)  # idempotency hits

    started_at = models.DateTimeField(null=True, blank=True)
    completed_at = models.DateTimeField(null=True, blank=True)
    error_message = models.TextField(blank=True)

    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['-created_at']
        indexes = [
            models.Index(fields=['student', 'status']),
            models.Index(fields=['status', 'created_at']),
        ]

        # One queued/running job per repo at a time. Re-installing the App
        # on the same repo while a backfill is still running just attaches
        # to the existing job rather than starting a duplicate.
        constraints = [
            models.UniqueConstraint(
                fields=['student_repo'],
                condition=models.Q(status__in=['queued', 'running']),
                name='one_active_backfill_per_repo',
            ),
        ]
```

### 2. Webhook hook

In `grading/services/github_app_events.py:handle_installation_repositories_event`
and the `handle_installation_event` paths, after a new `StudentRepo`
row is created, enqueue a backfill:

```python
def _enqueue_backfill(student_repo: StudentRepo, days_window: int = 90):
    """
    Idempotent enqueue. Called from both installation webhooks.
    Skips silently if a backfill already exists for this repo (caught
    by the unique constraint).
    """
    from grading.models import RepoBackfillJob

    try:
        job = RepoBackfillJob.objects.create(
            student_repo=student_repo,
            student=student_repo.user,
            days_window=days_window,
        )
    except IntegrityError:
        # Already queued or running — leave alone.
        return None

    # Hand off to async worker. Once Part 2 lands (Celery + Redis),
    # this becomes:  run_repo_backfill.delay(job.id)
    # Pre-Part-2: write a Django mgmt command + cron job that drains
    # the queue every 5 minutes.
    return job
```

### 3. Worker entry point

```python
# grading/services/repo_backfill_runner.py

def run_repo_backfill(job_id: int) -> None:
    """
    Consume a RepoBackfillJob: walk merged PRs in the window, run
    Layer 1 deterministic analysis on each, create Submission +
    GradingSession + DeterministicFinding + SkillObservation rows
    stamped with historical timestamps.

    Calls into the existing backfill_student_history flow with these
    arg overrides:
      - repo: from job.student_repo.full_name
      - user: job.student
      - cohort: job.student.cohort_membership.cohort
      - days: job.days_window
      - course: pick the cohort's first course (v1.1 routes by
        Course->Assignment binding once multi-opdracht ships)

    On success: status=DONE, completed_at=now, recompute
    SkillMetric.bayesian_score from the new observations.

    On failure: status=FAILED, error_message captured. Retry-safe —
    can be re-enqueued manually.
    """
```

### 4. Per-org configuration

Add fields on `Organization`:

```python
class Organization(models.Model):
    ...
    backfill_days_window = models.PositiveIntegerField(
        default=30,
        help_text=(
            'Days of history to auto-backfill when a student first '
            'connects a repo. Free tier: 30. Paid: 90 (configurable).'
        ),
    )
    auto_backfill_enabled = models.BooleanField(
        default=True,
        help_text='When False, students start with no historical data.',
    )
```

The webhook handler reads `student.organization.backfill_days_window`
when enqueueing. School admin can change it under Settings.

### 5. Frontend: 7-day grace-period banner

On the student dashboard (DashboardView), check whether the user has
any active or recently-completed RepoBackfillJob. If so, render a
banner above the dashboard content:

```
┌──────────────────────────────────────────────────────────────────┐
│ ⏳ LEERA is je werk aan het bekijken                               │
│ We scannen je laatste 90 dagen aan PR's en bouwen je skill graph │
│ op. Verwacht na 5+ PR's de eerste betrouwbare scores.             │
│ Voortgang: 12 / 47 PR's verwerkt.                                 │
└──────────────────────────────────────────────────────────────────┘
```

States:
- `queued` → "In de wachtrij — start binnenkort"
- `running` → progress bar with prs_processed / prs_found
- `done` (within last 7 days) → success banner that auto-dismisses
- `failed` → "We konden je geschiedenis niet ophalen — vraag je
  docent om handmatig te backfillen"

Banner dismisses automatically 7 days after job completion (or
immediately if user clicks ✕).

### 6. Teacher visibility

On the teacher's view of a student profile (TeacherStudentProfileView),
add a small badge near the Eindniveau widget:

- "Backfill: ✓ 47 PR's, 90 dagen" — green when complete
- "Backfill: bezig (12/47)" — orange when running
- "Backfill: nog niet uitgevoerd" — gray when no job exists (legacy
  students from before this feature landed)

Clicking the badge opens a modal with backfill details + a "Trigger
backfill nu" button that re-enqueues with the org's window setting.
Useful when a student transferred from another school and the docent
wants their old history pulled in.

## Cost + scaling

Per repo backfill (no LLM, Layer 1 only):
- ~60 PRs × 90 days = typical run
- ~3-5 GitHub API calls per PR (list + files + diff)
- ~150-300 API requests per backfill
- GitHub rate limit: 5000/hour authenticated → ~16 backfills/hour per token

For a school of 100 students all installing on day 1:
- ~6 hours to drain the queue at the rate limit
- The grace-period banner sets honest expectations

To accelerate: shard across multiple App installation tokens (one per
student install — already the case since LEERA App tokens are
per-installation). Effective rate limit becomes 5000 × N installs.
Probably enough for v1.1.

## Out of scope for v1.1

- **LLM-graded backfill.** Cost-prohibitive (€5+ per student × 100 =
  €500/month per cohort just for backfill). Layer 1 deterministic is
  enough to bootstrap the skill graph; live PRs from week 2 onward
  use the LLM as normal.
- **Cross-repo aggregation.** Each StudentRepo gets its own backfill.
  No merge-into-one-trajectory across repos in v1.1.
- **Forks / archived repos.** First install on a fork creates a new
  backfill job; original repo's history is the source of truth.
  Archived repos: skip silently (no API access).
- **Per-PR LLM upgrade tier.** Future v1.5 feature: school admin can
  pay extra to re-run LLM grading on a specific historical PR for
  end-of-year evaluation.

## Migration plan

1. New tables: `RepoBackfillJob`, fields on `Organization`.
2. Webhook handlers updated to enqueue jobs.
3. Worker registered in Celery (post Part 2). Pre-Part-2:
   `python manage.py drain_backfill_queue` cron job every 5 min.
4. Backfill all existing StudentRepos that don't have a completed
   job — one-time migration script with `--days=90`.
5. Frontend banner + teacher badge ship in the same release.

## Decision log

- **Why Layer 1 only?** Cost. €5-15 per LLM backfill × 100 students =
  prohibitive. Layer 1 deterministic gives 70% of the signal at 0%
  the cost. Schools can pay extra for LLM upgrade per-PR if needed.
- **Why per-org window?** Free tier should cap at 30 days to keep
  GitHub API costs predictable; paid tier opens to 90+ for
  competitive parity.
- **Why a Job model instead of just enqueueing a Celery task?**
  Visibility — students and teachers need to see "is the backfill
  done yet?". Pure Celery hides progress. Job rows are the source of
  truth, Celery just runs them.
- **Why grace-period banner instead of hiding the dashboard?**
  Hiding feels like LEERA is broken on day 1. Showing "we're learning"
  preserves the perception that LEERA is working hard for the student.
- **Why not auto-trigger on every push for missed history?** Push
  events only carry their own commits. Filling gaps would require
  full-history walk per push, which is wasteful. One-shot on install
  is enough.
