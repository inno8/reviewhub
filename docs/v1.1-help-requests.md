# v1.1 — Help Requests

**Status:** design draft. Not implemented. Ship AFTER `v1.1-multi-opdracht-model.md`
because help requests are scoped to an Assignment / StudentAssignment.
**Author:** discussion 2026-04-30
**Owner:** TBD

## Why

The student-facing Code Review page already has a "Request Help" button
on each Finding (FileReviewView.vue). Clicking it currently sets
`explanationRequested` on the Finding and changes the button label,
but **does nothing else**. The teacher never finds out.

The complete loop:

1. Student is stuck on a finding LEERA produced (the `suggested_code`
   isn't clear, the explanation didn't land, they don't understand
   *why* the original code is wrong).
2. Student clicks "Request Help" on the file review page.
3. The teacher who owns the project / course / assignment gets an
   in-app notification.
4. Teacher opens a "Hulpvragen" page that shows pending help requests,
   each with the full finding context (file + line + original_code +
   suggested_code + the LLM's explanation + which student).
5. Teacher resolves the help request (often by writing a personal
   explanation, or messaging the student, or marking it resolved
   after a 1:1 conversation).
6. Student sees their help request was resolved on their side.

## Hierarchy this lives inside

```
Cohort
  └─ Course (vak — has a teacher)
       └─ Assignment (opdracht)
            └─ StudentAssignment (student → repo)
                 └─ Submission (PR)
                      └─ Evaluation (per commit) — has Findings
                           └─ HelpRequest ← NEW
```

The teacher who receives the notification = `Course.teacher` (the
docent who owns this vak). For multi-teacher courses (rare in MBO v1)
all teachers in `Course.teachers` see the request.

## Models

```python
# evaluations/models.py (new)

class HelpRequest(models.Model):
    """
    A student asks for human help on a Finding the AI surfaced.
    Lives at the per-commit Code Review layer, not per-PR — students
    typically request help while still iterating on their code, before
    the docent has even started a formal review.
    """

    class Status(models.TextChoices):
        OPEN = 'open', 'Open'
        RESOLVED = 'resolved', 'Opgelost'
        DISMISSED = 'dismissed', 'Genegeerd'  # teacher closed without action

    finding = models.ForeignKey(
        'Finding',
        on_delete=models.CASCADE,
        related_name='help_requests',
    )
    student = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.CASCADE,
        related_name='help_requests',
    )
    # Denormalized for fast teacher-side queue queries — a docent's
    # "Hulpvragen" inbox should NOT need to walk
    # finding → evaluation → project → cohort_membership → cohort → course
    # on every load. Snapshot at creation; backfill if cohort/course
    # changes (rare).
    course = models.ForeignKey(
        'grading.Course',
        on_delete=models.CASCADE,
        related_name='help_requests',
        null=True, blank=True,
    )

    # Student's own framing — optional, but encouraged via UI:
    # "I tried X but Y, can you check Z?" gets better answers than
    # blank "ik snap dit niet".
    student_message = models.TextField(blank=True)

    status = models.CharField(
        max_length=16,
        choices=Status.choices,
        default=Status.OPEN,
    )

    # Teacher's response — written when resolving. May be empty if the
    # teacher resolved offline (1:1 conversation, in-class explanation).
    teacher_response = models.TextField(blank=True)
    resolved_by = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        on_delete=models.SET_NULL,
        null=True, blank=True,
        related_name='help_requests_resolved',
    )
    resolved_at = models.DateTimeField(null=True, blank=True)

    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['-created_at']
        indexes = [
            # Teacher's queue: open requests for courses I own, newest first.
            models.Index(fields=['course', 'status', '-created_at']),
            # Student's history: my own requests across all courses.
            models.Index(fields=['student', '-created_at']),
            # One open request per finding per student (idempotency).
            models.Index(fields=['finding', 'student', 'status']),
        ]
```

## API endpoints

```
POST   /api/help-requests/                  Student creates one
                                            Body: { finding_id, student_message? }
                                            Response: 201 with full HelpRequest
                                            Idempotent: if an open request already
                                            exists for (finding, student), return 200
                                            with the existing one.

GET    /api/help-requests/                  List
                                            Filters: ?status=open|resolved|all
                                                     ?course=<id>          (teacher)
                                                     ?student=<id>         (admin)
                                                     ?mine=true            (current user)
                                            Permissions: teacher sees their own
                                            courses; student sees their own requests;
                                            admin sees their org.

GET    /api/help-requests/<id>/             Detail with full Finding context
                                            (file_path, line, original/suggested
                                            code, explanation, evaluation/commit,
                                            student profile)

POST   /api/help-requests/<id>/resolve/     Teacher closes
                                            Body: { teacher_response?, status:
                                                    resolved|dismissed }

POST   /api/help-requests/<id>/reopen/      Student or teacher reopens a closed
                                            request (rare but useful when the
                                            "resolution" didn't actually help).
```

## Notification path

LEERA already has a `notifications` Django app (per CLAUDE.md). Reuse it:

```python
# When HelpRequest is created in serializer.create():
from notifications.models import Notification
teachers = course.teachers.all() if hasattr(course, 'teachers') else [course.teacher]
for teacher in teachers:
    if teacher and teacher.is_active:
        Notification.objects.create(
            user=teacher,
            kind='help_request',
            title=f"{student.username} vraagt hulp bij {finding.title[:50]}",
            body=finding.description[:200],
            link=f"/teacher/help-requests/{help_request.id}",
            related_object=help_request,
        )
```

In-app notification badge increment is automatic (existing
notifications system handles that). No email by default — most
docenten check LEERA frequently during a school day; email is noise.
Email can be a per-org / per-user setting in v1.2.

## Frontend

### Student side

**FileReviewView.vue** — already has a "Request Help" button. Wire it:

```typescript
async function requestExplanation() {
  if (!selectedFinding.value) return;
  actionLoading.value = true;
  try {
    const { data } = await api.helpRequests.create({
      finding_id: selectedFinding.value.id,
      student_message: helpMessageDraft.value,  // new optional textarea
    });
    selectedFinding.value.helpRequestId = data.id;
    selectedFinding.value.helpRequestStatus = data.status;
    helpMessageDraft.value = '';
    toast.success('Hulpvraag verstuurd naar je docent');
  } catch (e: any) {
    toast.error(e?.response?.data?.detail || 'Kon hulpvraag niet versturen');
  } finally {
    actionLoading.value = false;
  }
}
```

After requesting, the button becomes "✓ Hulpvraag verzonden" with a
secondary action "Open hulpvraag" linking to a thin
`/my/help-requests/<id>` page that shows the request + the teacher's
response (when resolved).

Add a small textarea (optional) above the button: "Wat heb je
geprobeerd? (optioneel)" — encourages reflection, gives the teacher
context, fits the LEERA pedagogy of behavioral proof.

### Teacher side

**New page: `/teacher/help-requests`** — `TeacherHelpRequestsView.vue`.

Two tabs: "Open" / "Opgelost".

Each row in the Open tab:

```
┌──────────────────────────────────────────────────────────────────┐
│ [avatar] Yusuf de Boer · Webdev · 2 hours ago                     │
│ "Scope method lacks input validation"                             │
│ app/Models/Book.php:27                                            │
│ ─                                                                 │
│ Yusuf's bericht: "Ik snap niet waarom ik throw moet gebruiken"   │
│                                                                   │
│ [Open hulpvraag →]                                                │
└──────────────────────────────────────────────────────────────────┘
```

Click "Open hulpvraag" → detail page with:

- Full finding context (the same FileReviewView side-by-side, but
  scoped to this one finding, no actions for the student)
- Yusuf's profile snapshot (skill graph in this category, last 5
  commits, current opdracht)
- A textarea for the teacher's response
- Two buttons:
  - **"Stuur antwoord en sluit"** → resolves with the response
  - **"Sluit zonder antwoord"** → dismisses (dropdown reason: "in
    class besproken", "in 1:1 besproken", "duplicate", "geen
    actie nodig")

Teacher's response (if any) is visible to the student on their own
help-request page. No DM-style threading — keep it simple. If a back-
and-forth is needed, that's a 1:1 conversation, not a LEERA workflow.

## Migration plan

Single migration: `evaluations/migrations/0XXX_help_request.py` —
creates the HelpRequest table, adds indexes. No backfill needed
(no historical data).

## Out of scope for v1.1

- Email notifications (default off; per-user setting comes later)
- DM-style threading (one teacher response per request, period)
- File attachments on the help request (not needed; the finding has
  the file context already)
- Help request analytics for school admins ("which findings get the
  most help requests" → which topics need re-teaching)
- Auto-suggest similar past help requests to the teacher when opening
  a new one (RAG over `Course.help_requests` filtered by the finding's
  category) — interesting but defer to v1.5

## Decision log

- **Why per-Finding, not per-PR?** Students typically need help
  *while iterating*, before any docent review of the PR. Help
  requests are about confusion on the AI's feedback, which is
  always per-finding.
- **Why not a generic comment thread on the finding?** Threading
  invites async chat which docenten don't have time for. One
  request, one response, closed. If more conversation needed, take
  it offline.
- **Why denormalize `course` on HelpRequest?** Hot read path in the
  teacher's inbox. Walking finding → evaluation → project → cohort
  membership → cohort → course on every poll is wasteful.
- **Why "dismissed" status separate from "resolved"?** "Resolved
  with response" and "closed without action" carry different
  pedagogical signals. Analytics can use the split.
