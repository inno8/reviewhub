# v1.1 — Full Dutch translation + i18n framework

**Status:** scoped, not started.
**Owner:** TBD.
**Trigger:** discovered May 2 2026 during pre-pitch demo audit. Roughly 13 demo-path views were translated to Dutch in commits `b15836e..33b3da4`. The other ~30 views (and email templates, API error messages) still have English strings.

## Problem

Leera ships as "Nakijken Copilot voor het MBO" — a Dutch product for Dutch
MBO-4 ICT teachers. Today, every user-facing string is hardcoded in `.vue`
templates, with no translation framework. The pre-pitch sprint translated
the demo path manually (~157 strings across 13 files), but:

- The rest of the app still shows English in roughly 30 views (cohort
  management, teacher student list, dev profile setup, ops dashboard,
  signup/onboarding flows, etc).
- Backend strings (notification email templates, password-reset emails,
  invite emails, API error responses) are entirely English.
- New features added without an i18n framework will keep regressing —
  hardcoding English is the path of least resistance for any developer
  shipping fast.
- We can't offer a language toggle to schools that want bilingual staff
  views (some MBO schools have international students or partner orgs).

A full hand-translation without tooling is also fragile — terminology
drift across files, no way to track coverage, no way to lint regressions.

## Goal

Every user-facing string in Leera (frontend templates, backend email
templates, API error messages surfaced to users) lives in a translation
catalogue. Default locale is `nl-NL`. English is the fallback for legal
copy and any string we haven't translated yet.

CI prevents new hardcoded English from landing.

## Scope

### Frontend
- Add `vue-i18n@9` to `frontend/package.json`. Set up `createI18n` in
  `main.ts` with `locale: 'nl-NL'`, `fallbackLocale: 'en'`,
  `legacy: false`, `globalInjection: true`.
- Create `frontend/src/locales/nl.json` and `frontend/src/locales/en.json`.
- Run `vue-i18n-extract` to extract all `$t()` and `<i18n-t>` references
  to a coverage report.
- For uncovered strings: manually replace hardcoded `<template>` strings
  with `{{ $t('key.path') }}`. Group keys by view: `dashboard.title`,
  `dashboard.next_up.label`, etc.
- Native-Dutch review pass on `nl.json`. Get a Dutch ICT teacher to walk
  the file and flag terminology that reads stiff or wrong (e.g. "review"
  vs "nakijken" — the rest of the product uses "nakijken", so be
  consistent).
- Add ESLint rule `vue-i18n/no-raw-text` to catch new hardcoded strings
  in templates. Must be `error` level so CI blocks PRs.

### Backend
- Django: enable `LocaleMiddleware`, set `LANGUAGE_CODE = 'nl-NL'`.
- Wrap all user-facing API error strings with `gettext_lazy` (`_("...")`).
- Convert all email templates (`*.html` and `*.txt` under
  `django_backend/*/templates/`) to use `{% load i18n %}` + `{% trans %}`.
- Run `manage.py makemessages -l nl_NL` to generate `.po` files.
- Translate the `.po` catalogue.
- Run `manage.py compilemessages` in CI.

When extracting strings, lift the Dutch from these files first — they're
the canonical voice.

## Out of scope (v1.2+)

- Multi-locale support beyond Dutch + English fallback. The Dutch
  market is the v1 focus; if v2 expands to other EU MBO equivalents
  (Flanders, Germany), revisit then.
- Translation Memory tooling (Lokalise, Weblate, etc) — overkill for
  one-locale-at-a-time scale.
- Right-to-left support — irrelevant for Dutch.

## Why this matters

A docent who sees "Settings" instead of "Instellingen" reads the product
as "this isn't really for us, this is an American tool with a Dutch
sticker on it." The demo path is fine. The cohort management page they
land on Monday morning to set up next week's klas? Still half English.
That's where credibility is lost — when the curiosity tour ends and the
real work begins.
